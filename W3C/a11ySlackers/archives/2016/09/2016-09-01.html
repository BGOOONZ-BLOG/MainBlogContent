    [dna] hey slackers, anyone have any guidance on best practices for loading a page with an important notice on it?
  38. [dna] i know of role=alert, but is that intended for client-side rendered notices only?
  39. [dna] the other alternative I see is perhaps redirecting keyboard focus on load to a relevant div filled with information
  40. zakim-robot
    @zakim-robot
    Sep 01 17:41
    [michiel] I would advice the alert role.
  41. [michiel] http://w3c.github.io/aria-practices/#alert
  42. zakim-robot
    @zakim-robot
    Sep 01 17:55
    [marcysutton] I've used it on server-side rendered alerts as well
  43. [marcysutton] but there might be a timing issue...you'd have to test it
  44. [marcysutton] if you put the notice early enough in the source order it should be discoverable
  45. [marcysutton] (even without role=alert)
  46. [dna] @marcysutton what do you mean by a timing issue?
  47. [marcysutton] Typically when you use ARIA live you establish the region at load time, then append messages to it. If the message and the role are rendered at the same time, I would expect the message might get missed and not announced
  48. [marcysutton] but that's just a hunch
  49. [dna] ah gotcha
  50. [karlgroves] That’s right. Adding the aria live properties later won’t work. Screen readers keep a copy of the DOM in a buffer. The only things that trigger buffer refresh are adding/ removing items from the DOM or changing display properties
  51. [karlgroves] modifying element attributes won’t do it
  52. zakim-robot
    @zakim-robot
    Sep 01 18:01
    [dna] @karlgroves so that means the role=alert won’t work, right? would redirecting keyboard focus to a tabindexed div with content be too invasive?
  53. [karlgroves] It would work fine as long as it exists ahead of time or if the DOM is modified somehow
  54. [karlgroves] So, if you load the page with role=alert it should be announced as soon as the speech buffer allows it
  55. [dna] oh, great
  56. [dna] i’ll try it out
  57. [dna] second question that’s unrelated but arguably more important: does anyone know if google penalizes screen-reader-only text as “cloaking”?
  58. [dna] or if that’s on their radar

  59. [karlgroves] On page load, the first things read are things like the page title, URL, and other characteristics about the page (# of links etc.)

    If you have a role=alert on page load it should read that content as soon as the other stuff is done with.

  60. [marcysutton] if the text is spammy and unrelated to the rest of the content maybe?

  61. [karlgroves] > does anyone know if google penalizes screen-reader-only text as “cloaking”?

    Generally, no. Of course it is is too long and resembles keyword stuffing then yes

  62. [dna] yeah, i figure a comma separated list would be a good heuristic
  63. [karlgroves] Hidden text like validation messages are OK. Loads of people do that, including Google themselves.
  64. [dna] true, good point
  65. [karlgroves] But not
