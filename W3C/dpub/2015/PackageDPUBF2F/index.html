What is a "Package"?

A packaging mechanism is used to associate the components of a work or multiple works to one another in a meaningful way for transport and distribution.

The mechanism must identify a sequence for the content and indicate which components are to be included in the package. There are defined packaging mechanisms that offer efficiencies and improved UX. We have created packaging use cases, including the basic packaging requirements of EPUB.

What are the basic criteria that a package must meet?

  1. Robust metadata
  2. Package available online and offline
  3. Access to components without traversing entire package
  4. Enable update/refresh of components that are new or have changes (and not others)

What are the options for packaging?

  • Zip and defined derivatives
  • Multipart MIME and similar
  • Databasing (out-of-scope)

ZIP

Zip is stable and ubiquitous. Many tools exist to support its use. The Central Directory, which provides information about how to read the file, is the last file in the package. This means that tools that open zip files must traverse the entire package in order to understand how to access any of its members. The spec allows for limited metadata.

Multipart MIME

Multipart MIME (RFC 1521 and RFC 1341) allows for inclusion of non-ASCII content in messages. The components are in a tree structure, using the protocols that are the foundation of email and HTTP.

  • Header has a multipart content-type
  • Parts are encapsulated by boundaries
  • Each part has its own header and body
  • Content-type can carry a subtype, including "mixed", "alternative", and "related"

Multipart MIME cotinued

In 1999, OEBFF, the predecessor to EPUB, tried this out. The spec uses Content-Type:multipart/related, indicating that sub-parts form an aggregate whole.

Packaging on the Web

Packaging on the Web takes the foundational principles of MPM but creates a new Content-type:
Content-type:application/package

  • Each component has a boundary with an HTTP syntax
  • Each part must include content-locations (relative URI, with location of package as base - leading to the ID and Frag IDs for package and components)
  • Parts that share content-location but differ in content-type offer content negotation
  • HTML or HTTP link means that a UA may download and cache whole package in one request. Then, resolve relative URIs to the original page by getting the content from the (downloaded) package
